# Database Migrations

This directory contains Alembic database migrations for the Face Recognition Attendance System.

## Quick Start

### Initialize Database
```bash
# Apply all migrations
alembic upgrade head

# Or use the db_manager script
python scripts/db_manager.py init
```

### Seed Test Data
```bash
python scripts/db_manager.py seed
```

## Creating Migrations

### Auto-generate from Model Changes
```bash
# After modifying models in app/models/
alembic revision --autogenerate -m "description_of_changes"

# Review the generated migration in migrations/versions/
# Edit if necessary, then apply:
alembic upgrade head
```

### Create Empty Migration
```bash
alembic revision -m "custom_change"
```

## Migration Commands

### Upgrade
```bash
# Upgrade to latest
alembic upgrade head

# Upgrade by 1 revision
alembic upgrade +1

# Upgrade to specific revision
alembic upgrade abc123
```

### Downgrade
```bash
# Downgrade by 1 revision
alembic downgrade -1

# Downgrade to specific revision
alembic downgrade abc123

# Downgrade all (back to empty database)
alembic downgrade base
```

### Information
```bash
# Show current revision
alembic current

# Show migration history
alembic history

# Show pending migrations
alembic history --verbose
```

## Best Practices

### Before Creating Migrations
1. **Review your model changes** - Ensure all changes are intentional
2. **Test locally first** - Apply migrations in development
3. **Review generated SQL** - Check the migration file before committing

### Writing Migrations
1. **Be specific** - Use descriptive migration messages
2. **Test both ways** - Ensure both upgrade() and downgrade() work
3. **Handle data** - Include data migration if schema changes affect existing data
4. **No manual edits after deployment** - Never modify deployed migrations

### Example: Adding a Column with Data Migration
```python
def upgrade():
    # Add column
    op.add_column('users', sa.Column('verified', sa.Boolean(), nullable=True))
    
    # Migrate existing data
    op.execute("UPDATE users SET verified = true WHERE is_active = true")
    
    # Make non-nullable after data migration
    op.alter_column('users', 'verified', nullable=False)

def downgrade():
    op.drop_column('users', 'verified')
```

## Production Workflow

### Development
```bash
# 1. Make model changes
# 2. Generate migration
alembic revision --autogenerate -m "add_user_verification"

# 3. Review and test
alembic upgrade head

# 4. Test rollback
alembic downgrade -1
alembic upgrade head

# 5. Commit to git
git add migrations/versions/...
git commit -m "migration: add user verification"
```

### Staging/Production
```bash
# 1. Pull latest code
git pull

# 2. Backup database
pg_dump attendance_db > backup_$(date +%Y%m%d).sql

# 3. Apply migrations
alembic upgrade head

# 4. Verify
alembic current

# 5. Test application
curl http://localhost:8000/health
```

## Troubleshooting

### Migration conflicts
If multiple developers create migrations:
```bash
# Merge migrations
alembic merge <rev1> <rev2> -m "merge migrations"
```

### Stuck at wrong revision
```bash
# Stamp database at current state (use carefully!)
alembic stamp head
```

### Reset development database
```bash
# Drop everything and start fresh
python scripts/db_manager.py reset
alembic upgrade head
python scripts/db_manager.py seed
```

## PostgreSQL vs SQLite

The default configuration uses SQLite for development. For production with PostgreSQL:

1. Update `alembic.ini`:
```ini
sqlalchemy.url = postgresql+asyncpg://user:password@localhost/attendance_db
```

2. Or use environment variable:
```bash
export DATABASE_URL="postgresql+asyncpg://user:password@localhost/attendance_db"
```

## CI/CD Integration

Migrations are automatically tested in CI (`.github/workflows/test.yml`):
- Migrations are applied to a test database
- Schema is validated with `alembic check`

## Notes

- **Async Support**: This project uses async SQLAlchemy, migrations run in async mode
- **Auto-generate Limitations**: Always review autogenerated migrations, they may miss:
  - Index changes
  - Constraint modifications
  - Data type changes (especially custom enums)
- **Version Control**: All migrations must be committed to git
